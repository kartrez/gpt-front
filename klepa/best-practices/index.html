Как работать с Клёпой эффективно

Клёпа лучше всего работает, когда вы подбираете режим под конкретную задачу. Не пытайтесь решать всё одним и тем же способом: сначала спланируйте, затем уточните требования, после кодируйте, а для проверки используйте Debug и Review.

Правильный выбор режима уменьшает количество итераций, делает ответы точнее и помогает быстрее довести задачу до рабочего результата.

Режимы и когда их использовать

Architect
Plan and design before implementation

Режим предварительного проектирования. Помогает сформировать архитектуру, выбрать подход и разложить задачу до начала написания кода.

Когда использовать:
  • нужно продумать структуру новой функции или сервиса
  • требуется оценить несколько вариантов решения
  • нужно зафиксировать ограничения и допущения
Пример: Сначала используйте Architect, чтобы получить план, затем переходите в Code.
Code
Write, modify, and refactor code

Основной режим для написания, изменения и рефакторинга кода. Лучше всего работает, когда задача уже понятна и есть чёткий контекст.

Когда использовать:
  • нужно реализовать готовую спецификацию
  • требуется внести точечные изменения
  • нужно отрефакторить участок кода без смены логики
Пример: После плана и уточнений переключайтесь в Code для реализации конкретных файлов.
Ask
Get answers and explanations

Режим для вопросов, объяснений и проверки понимания. Используйте его, когда нужно разобрать концепцию без немедленной правки кода.

Когда использовать:
  • нужно понять, как работает библиотека или API
  • требуется объяснение архитектурного решения
  • нужно быстро уточнить термин или подход
Пример: Спросите в Ask, если хотите понять причины ошибки или выбор архитектуры.
Debug
Diagnose and fix software issues

Режим для поиска причины ошибки, воспроизведения проблемы и подготовки исправления. Подходит, когда уже есть симптом и нужно найти источник сбоя.

Когда использовать:
  • приложение падает или ведёт себя нестабильно
  • нужно найти причину некорректного результата
  • требуется проверить логи, трассировку или окружение
Пример: Если код работает не так, как нужно — переключайтесь в Debug до внесения новых изменений.
Orchestrator
Coordinate tasks across multiple modes

Режим-координатор для сложных задач. Помогает распределять работу между Architect, Code, Ask, Debug и Review.

Когда использовать:
  • задача большая и многошаговая
  • нужно вести работу в нескольких режимах подряд
  • требуется удерживать общий план и последовательность действий
Пример: Используйте Orchestrator как диспетчер: он помогает не потерять структуру задачи.
Review
Review code changes locally

Режим локальной проверки изменений. Полезен после реализации, когда нужно оценить качество кода, найти недочёты и подтвердить готовность к merge.

Когда использовать:
  • нужно проверить diff перед отправкой
  • важно найти мелкие ошибки и регрессии
  • требуется финальная оценка качества изменений
Пример: После Code и Debug завершайте цикл в Review, чтобы не пропустить проблемы до релиза.
